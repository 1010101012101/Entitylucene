If Windows still will not boot, let me know and we can try to fix the second problem.  h2Expanation/h2  There are two partition schemes that are important here. They are the emMaster Boot Record/em (MBR) partitioning scheme and the emGlobally Unique Identifier/em (GUID) partitioning scheme. The MBR scheme contains a emMBR Partition Table/em which allows for the definition of only 4 partitions. The GUID scheme contains the same 4 entry emMBR Partition Table/em and a emGUID Partition Table/em (GPT) whose length is determined at time of creation. This value is typically much larger than 4 entries.  When a disk is partitioned using the GUID scheme, only the first entry is used in the MBR table. On your, disk this table contained the following data.  precodeDisk: /dev/disk1    geometry: 121601/255/63 [1953525168 sectors] Signature: 0xAA55          Starting       Ending  #: id  cyl  hd sec -  cyl  hd sec [     start -       size] ------------------------------------------------------------------------  1: EE    0   0   2 - 1023 254  63 [         1 - 1953525167] &lt;Unknown ID&gt;  2: 00    0   0   0 -    0   0   0 [         0 -          0] unused        3: 00    0   0   0 -    0   0   0 [         0 -          0] unused        4: 00    0   0   0 -    0   0   0 [         0 -          0] unused       /code/pre  The id of EE indicates the disk is GUID partitioned and the GPT should be used to determine partitioning. The next six columes (Starting/Ending cyl/hd/sec) exist for legacy purposes. Experience has taught me that on Mac computers, it does not matter what values are in these six columns. In your case, the first three contain the lowest possible emvalid/em values and the next three contain the highest possible emvalid/em values. What is important is the start and size columns. Your disk is divided into 1953525168 sectors where each sector is 512 bytes. The first sector is numbered 0 and contains the MBR which also contains the MBR table. The partition with the EE id is defined to cover the rest of the disk. This gives older legacy partitioning software the impression all space has been allocated on the disk. This is why you will see the phrase emProtective Master Boot Record/em (PMBR) in other documentation.   Below is a partial list of the contents of the GPT on your disk.  precode       start        size  index  contents            0           1         PMBR            1           1         Pri GPT header            2          32         Pri GPT table           34      262144      1  GPT part - E3C9E316-0B5C-4DB8-817D-F92DF00215AE       262178        2014                264192      409600      2  GPT part - C12A7328-F81F-11D2-BA4B-00A0C93EC93B       673792  1751953408      3  GPT part - 53746F72-6167-11AA-AA11-00306543ECAC   1752627200     1269760      4  GPT part - 426F6F74-0000-11AA-AA11-00306543ECAC   1753896960      409600            1754306560   199217152      5  GPT part - EBD0A0A2-B9E5-4433-87C0-68B6B72699C7   1953523712        1423            1953525135          32         Sec GPT table   1953525167           1         Sec GPT header /code/pre  Each table entry is 128 bytes long. Note, there is a backup table/header at the end of the disk. The emMicrosoft Reserved/em partition does not belong and should be removed. The emEFI/em partition usually occurs first. It is not used to boot the OS X operating system, but is required by OS X for other uses. The emApple_CoreStorage/em partition contains most of your OS X "Macintosh FD" volume. The emApple_Boot/em partition (I believe) contains your "Bootable Recovery Partition". The software, contained in this partition, is the replacement for the recovery software that was on the installation DVD. Finally, emMicrosoft Basic Data/em partition contains your Windows volume.  On your Mac, Windows requires a MBR scheme which is a conflict since OS X uses a GUID scheme. The following changes were necessary to the MBR partition table to fool Windows in to thinking your computer is using a MBR scheme.  ol liThe size of the EE partition was adjust to just cover the emEFI/em partition./li liData was copied from the GPT to the MBR partition table. The was done for first three partitions that followed the emEFI/em partition. In your case, this was the rest of the partitions. The start and size were copied exactly. Each GUID was translated to an id. I would give a translation table, but unfortunately not all the translations are one-to-one./li liThe windows partition was marked as active. Active means boot to this partition. When printing the MBR partition table, the active partition is marked with an * character./li /ol  When corrected the MBR partition table should appear as follows.  precodeDisk: /dev/disk1    geometry: 121601/255/63 [1953525168 sectors] Signature: 0xAA55          Starting       Ending  #: id  cyl  hd sec -  cyl  hd sec [     start -       size] ------------------------------------------------------------------------  1: EE    0   0   2 - 1023 254  63 [         1 -     673791] &lt;Unknown ID&gt;  2: AC 1023 254  63 - 1023 254  63 [    673792 - 1751953408] &lt;Unknown ID&gt;        3: AB 1023 254  63 - 1023 254  63 [1752627200 -    1269760] Darwin Boot      *4: 07 1023 254  63 - 1023 254  63 [1754306560 -  199217152] HPFS/QNX/AUX /code/pre  The command that was used to make the changes to this table on your disk is called codefdisk/code.
